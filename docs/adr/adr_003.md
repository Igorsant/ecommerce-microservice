# ADR 003: Comunicação entre Microsserviços

**Status:** Aceito  
**Data:** 2026-04-24  
**Autores:** Time de Arquitetura

---

## Contexto

O sistema é composto por cinco microsserviços independentes, cada um com seu próprio banco de dados:

| Serviço          | Stack                        | Banco de Dados   |
|------------------|------------------------------|------------------|
| auth-service     | Node.js / TypeScript (Prisma) | PostgreSQL       |
| user-service     | Node.js                      | PostgreSQL       |
| product-service  | Node.js                      | PostgreSQL       |
| order-service    | Elixir / Phoenix (Ecto)      | PostgreSQL       |
| payment-service  | Node.js (Knex)               | PostgreSQL       |

Cada serviço é dono exclusivo de seus dados. Nenhum serviço acessa diretamente o banco de outro. É necessário definir como eles se comunicam e como garantir consistência de dados entre eles.

---

## Decisão

A comunicação entre serviços será feita via **HTTP síncrono (REST)** para operações que exigem resposta imediata (consultas de dados) e via **eventos assíncronos** para propagação de estado após a conclusão de uma operação (criação de entidades derivadas).

Os três fluxos de comunicação do sistema são:

### 1. auth-service → user-service

**Gatilho:** Registro de um novo usuário.

**Fluxo:**
1. O `auth-service` recebe as credenciais (`email`, `password`) e cria um registro na tabela `users` com o hash da senha.
2. Após persistir com sucesso, o `auth-service` notifica o `user-service`, passando o `user_id` gerado.
3. O `user-service` cria o perfil do usuário na sua própria tabela `users` com os dados de identificação (`name`, `phone`), referenciando o mesmo `user_id`.

**Schemas envolvidos:**

```
auth-service.users
  id            TEXT (UUID) PK
  email         VARCHAR(255) UNIQUE
  password_hash VARCHAR(255)
  created_at    TIMESTAMP

user-service.users
  id         SERIAL PK       ← referencia lógica ao auth-service.users.id
  name       TEXT
  phone      TEXT
  created_at TIMESTAMP
```

**Responsabilidade:** O `auth-service` é dono das credenciais; o `user-service` é dono dos dados de perfil. O `user_id` é o elo entre os dois domínios.

---

### 2. order-service → product-service

**Gatilho:** Criação de um pedido.

**Fluxo:**
1. O `order-service` recebe a requisição de criação de pedido com a lista de produtos e quantidades.
2. Para cada item, o `order-service` consulta o `product-service` via HTTP para obter o preço atual e validar a disponibilidade em estoque.
3. O `order-service` persiste o pedido com o `total_amount` calculado e os itens com o preço no momento da compra (`order_items.price`).
4. Após confirmar o pedido, o `order-service` notifica o `product-service` para decrementar o estoque via `PATCH /products/:id/stock`.

**Schemas envolvidos:**

```
order-service.orders
  id           UUID PK
  user_id      UUID           ← referencia lógica ao auth-service.users.id
  status       STRING         (pending | ...)
  total_amount DECIMAL
  inserted_at  TIMESTAMP

order-service.order_items
  id         UUID PK
  order_id   UUID FK → orders.id
  product_id UUID           ← referencia lógica ao product-service.products.id
  quantity   INTEGER
  price      DECIMAL        ← snapshot do preço no momento do pedido

product-service.products
  id          UUID PK
  name        VARCHAR(255)
  description TEXT
  price       DECIMAL(10,2)
  category    VARCHAR(50)

product-service.stock
  id         UUID PK
  product_id UUID FK → products.id UNIQUE
  quantity   INTEGER
```

**Responsabilidade:** O `order-service` captura um snapshot do preço em `order_items.price` para garantir imutabilidade histórica dos pedidos. A atualização de estoque é de responsabilidade do `product-service`, acionado pelo `order-service`.

---

### 3. payment-service → order-service

**Gatilho:** Solicitação de pagamento de um pedido.

**Fluxo:**
1. O `payment-service` recebe a requisição de pagamento com o `order_id` e o `amount`.
2. O `payment-service` consulta o `order-service` para validar que o pedido existe e que o valor confere com `orders.total_amount`.
3. O `payment-service` cria o registro de pagamento com status `pending` e processa o pagamento.
4. Ao concluir o processamento, o `payment-service` atualiza seu próprio status (`approved` ou `refused`) e notifica o `order-service` para que ele atualize `orders.status` conforme o resultado.

**Schemas envolvidos:**

```
payment-service.payments
  id         UUID PK
  order_id   UUID           ← referencia lógica ao order-service.orders.id
  status     ENUM           (pending | approved | refused)
  amount     DECIMAL(10,2)
  created_at TIMESTAMP

order-service.orders
  id           UUID PK
  status       STRING         ← atualizado pelo payment-service após o pagamento
  total_amount DECIMAL
```

**Responsabilidade:** O `payment-service` é dono do ciclo de vida do pagamento. O `order-service` é responsável por refletir o resultado financeiro no status do pedido.

## Diagrama de Comunicação

```
┌─────────────────┐   POST /users (user_id)   ┌──────────────────┐
│   auth-service  │ ────────────────────────► │   user-service   │
│  users          │                           │  users           │
│  (credentials)  │                           │  (profile)       │
└─────────────────┘                           └──────────────────┘

┌─────────────────┐  GET /products/:id        ┌──────────────────┐
│  order-service  │ ◄──────────────────────── │  product-service │
│  orders         │                           │  products        │
│  order_items    │  PATCH /products/:id/stock│  stock           │
│                 │ ────────────────────────► │                  │
└─────────────────┘                           └──────────────────┘

┌─────────────────┐  GET /orders/:id          ┌──────────────────┐
│ payment-service │ ────────────────────────► │  order-service   │
│  payments       │                           │  orders          │
│                 │  PATCH /orders/:id/status │                  │
│                 │ ────────────────────────► │                  │
└─────────────────┘                           └──────────────────┘
```

## Consequências

### Positivas
- Cada serviço mantém soberania sobre seus dados — sem acesso cruzado a bancos de dados.
- O snapshot de preço em `order_items.price` garante integridade histórica dos pedidos independente de mudanças de preço no `product-service`.
- A separação de credenciais (auth) e perfil (user) reduz a superfície de exposição de dados sensíveis.

### Negativas / Trade-offs
- Consistência eventual: uma falha entre a criação do `auth_user` e a notificação ao `user-service` pode deixar um usuário sem perfil. É necessário um mecanismo de compensação (retry, outbox pattern) para esse fluxo.
- O `order-service` depende da disponibilidade do `product-service` em tempo de criação do pedido (acoplamento temporal no fluxo síncrono).
- O estado de `orders.status` é derivado do `payment-service`, o que exige que o `order-service` seja idempotente na atualização de status.

### Riscos Identificados
- Divergência de `user_id` entre `auth-service` e `user-service` caso o `user-service` use `SERIAL` (inteiro) como PK e o `auth-service` use `TEXT` (UUID). Recomenda-se alinhar os tipos de identificador ou adicionar uma coluna `auth_id` explícita no `user-service`.
